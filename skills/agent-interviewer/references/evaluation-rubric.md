# 面试证据与评价量表

## 原则

评价对象是本场回答提供的证据，不是候选人的人格。**未覆盖不等于能力弱**；弱证据表示问过但回答不足，未覆盖表示本场没有形成判断条件。表达不流畅本身不等于不可信，只有同时出现细节缺失、不可核验声明或矛盾时，才提高真实性风险。

不要在模拟过程中展示账本、评分或判断。

## 私有证据账本

对关键声明记录：

| 字段 | 内容 |
|---|---|
| Claim | 候选人声称亲自完成或产生的结果 |
| Evidence | 职责、决策、代码/链路、规模、基线、指标口径、日志、实验、失败案例 |
| Risk | 模糊主语、术语堆砌、无基线数字、过度完美、答非所问 |
| Contradiction | 与先前回答、简历或约束不一致之处，以及是否得到合理解释 |
| Coverage | 强证据 / 一般证据 / 弱证据 / 未覆盖 |
| Confidence | 高 / 中 / 低；说明由几条独立回答支撑 |

可信证据通常能跨层连接“为什么做 → 如何实现 → 如何验证 → 哪里失败 → 如何改进”。只会给定义不是项目证据；能诚实说明没做过并界定边界，通常比编造细节风险更低。

## 背稿与真实性判断

背稿信号必须绑定具体回答，不可凭语气臆断：

- 连续给出完整术语清单，却无法解释其中一个模块的数据流；
- 指标漂亮，但说不清基线、样本、口径或归因；
- 始终使用“我们”，无法隔离个人决策和代码；
- 一遇反事实或故障场景，答案从具体项目退回通用模板；
- 前后数字或架构边界冲突，且无法合理解释环境/口径差异。

反证包括：能指出自己方案的缺陷、记得具体失败、区分当前实现和理想方案、在不知道时明确说不知道。

## 能力评分

分别评价：项目归属与可信度、Agent/RAG 深度、工程落地、系统设计、模型基础、问题分析、表达沟通。

| 分数 | 锚点 |
|---:|---|
| 1 | 已覆盖，但核心概念或实践判断明显错误，无法在澄清后修正 |
| 2 | 有局部知识或参与经历，依赖提示，缺少独立闭环证据 |
| 3 | 达到目标级别基本要求；能讲清实现、取舍和验证，存在可控缺口 |
| 4 | 深度和工程判断稳定；能处理失败、反事实与跨模块权衡 |
| 5 | 超出目标级别；能定义问题边界、演进路线和组织/业务级取舍 |
| 未覆盖 | 本场没有足够问题或回答，禁止折算成 0/1 分 |

评分必须附一条以上本场证据。目标级别改变时，同一回答的岗位判断可改变，但事实证据不变。

## 通过概率与匹配度

通过概率是面试官基于本场表现的区间判断，不是客观预测：

- 使用 10% 或更粗粒度边界，如 `50%–70%`；不要写 `63%`。
- 覆盖充分、证据一致时可以缩窄；提前结束、缺 JD 或关键能力未覆盖时扩大区间。
- 写明 2–4 个最可能改变区间的变量，例如目标级别、编码轮表现或项目指标核验。
- 不因单个亮点给出极高概率，也不因单个“不会”直接判零。

岗位匹配度使用：强匹配 / 可匹配 / 偏弱 / 不匹配，并注明是针对哪类岗位和级别。若 JD 缺失，只能给方向匹配，不能假装完成岗位评估。

## 最终报告模板

### 1. 面试结论

给出建议（通过 / 倾向通过 / 保留 / 倾向不通过 / 不通过）、目标岗位与级别、三条关键依据。区分候选人原话事实与面试官推断。

### 2. 项目可信度

逐个核心项目列出可信部分、待核验声明、矛盾是否解释、总体置信度。

### 3. 背稿迹象

列出具体回答的转述、为何像背稿，以及可能的替代解释。没有足够证据时写“未发现明显迹象”，不要强行挑刺。

### 4. 工程细节缺口

按影响排序列出缺失的规模、基线、指标、失败处理、上线与运维证据。

### 5. 能力分项

每项给 `1–5 / 未覆盖`、证据强度、依据和目标级别差距。

### 6. 通过概率与岗位匹配

给区间、置信度、岗位匹配度，以及会改变判断的变量。

### 7. 风险点

只写与招聘决策相关且有证据的风险；将“事实”和“推断”分栏。

### 8. 改进建议

按优先级给 3–5 项可执行建议，优先补项目证据和工程闭环，再补知识面。

### 9. 回答框架

只针对本场弱回答提供结构、关键验证点和示例思路。不得伪装成唯一标准答案，也不要替候选人编造没做过的经历。
