# Agent / 大模型应用开发面试模式

研究日期：2026-07-06

## 使用边界

本文件提炼面试结构和能力域，不是题库。下面的比例是用于启动模拟的**启发式**，不是统计分布或真实出现频率。公开面经有转载、营销、幸存者偏差和岗位混杂；搜索结果数量也不是统计证据。小红书等存在登录墙、搜索不稳定的平台，只有能直接核验原帖时才纳入，不能从二手摘要虚构“高频”。

每次面试优先服从当前 JD、级别、简历和用户要求。只有用户要求更新面经时才联网修订本文件；日常模拟不因临时搜索结果改变规则。

## 默认流程权重

| 阶段 | 默认占比 | 目的 |
|---|---:|---|
| 开场与经历定位 | 5%–10% | 校准表达、经历主线和岗位动机 |
| 简历项目深挖 | 50%–65% | 验证本人贡献、深度、效果和真实性 |
| Agent / LLM 专项 | 15%–25% | 检查能否从框架名进入机制与工程取舍 |
| 工程、基础、系统设计或编码 | 10%–20% | 验证软件工程底座与岗位缺口 |
| 候选人反问 | 5%–10% | 观察岗位判断和沟通质量 |

比例不需要机械凑满。项目回答已经覆盖某个专项时，不重复问八股；面试时长缩短时先保留项目真实性、岗位核心能力和一个工程反事实。

## 岗位路由

| 岗位 | 提高权重 | 不应忽略 |
|---|---|---|
| Agent 开发 | Agent loop/workflow、state、memory、tool/MCP、权限、终止、评测、trace | 后端可靠性、模型边界、成本和安全 |
| LLM 应用工程 | RAG 全链路、prompt/context、模型选择、评测、业务指标、实验迭代 | 数据治理、服务化、失败降级和可观测 |
| AI 后端 | 网关、并发、队列、缓存、限流、隔离、部署、SLA、成本 | RAG/Agent 基本机制及输出质量评测 |

训练算法岗不是本 Skill 的默认目标。若 JD 明确要求微调、推理优化或训练，应提高 Transformer、PEFT/LoRA、对齐、量化、推理服务和算力分析的比重。

## 级别路由

| 级别 | 合格证据 |
|---|---|
| 初级 | 能说清自己实现的链路、接口、数据与调试过程；知道方案限制；不把 demo 描述成生产系统 |
| 中级 | 能独立拆解问题、比较方案、定义指标、处理失败与上线约束；能用具体证据说明个人贡献 |
| 高级 | 能确定边界和演进路线，处理跨团队、成本、SLA、安全与组织约束；能解释取舍的长期代价和业务结果 |

## 项目深挖维度

按回答选择一个维度继续，不要一轮全部询问：

- 问题与约束：为何值得用 LLM/Agent，而不是确定性流程？
- 个人边界：本人做了哪些决策、代码和实验？团队成果是什么？
- 架构与数据流：状态在哪里，模块如何通信，失败如何传播？
- 规模与基线：请求量、语料量、工具数、上下文长度、原方案表现。
- 指标与口径：离线集、线上指标、judge/人工评测、实验切分和回归。
- 失败与恢复：超时、幻觉、脏数据、循环、工具失败、依赖降级。
- 选型与反事实：替代方案、为何不选、规模十倍或预算减半时如何变化。
- 生产证据：trace、日志、告警、事故、成本、延迟、权限与审计。

## 专项能力地图

- **RAG：** ingest、解析、chunk、metadata/ACL、embedding、稀疏/稠密召回、rerank、context 组装、引用、评测、更新删除。
- **Agent：** workflow 与开放 loop、planning、state、memory、tool schema、终止条件、重试/熔断、human-in-the-loop、多 Agent 的必要性。
- **Tool / MCP：** 发现与注册、schema 兼容、权限和风险等级、幂等、超时、错误语义、可见性收缩、审计。
- **评测：** 任务成功、轨迹/工具选择、检索与生成分层指标、bad case fixture、回归集、线上反馈、漂移。
- **工程化：** 并发、队列、缓存、模型网关、配额、可观测、灰度、降级、成本和 p95/p99。
- **安全：** prompt injection、数据泄漏、租户隔离、工具副作用、内容安全、审批和审计。
- **模型基础：** token/context、采样、embedding、attention/KV cache、微调与 RAG 边界、推理部署；深度随 JD 调整。

## 更新规则

新增面经先归档来源，再提炼模式。只有出现多个可区分来源或目标 JD 明确变化时才调整默认权重。单篇新帖子可以扩充问题维度，但不能单独制造“行业普遍流程”的结论。
